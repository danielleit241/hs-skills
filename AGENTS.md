# Repository Guidelines

## Project Structure & Module Organization

This repository ships the **HS Skills** toolkit for Claude Code and Codex. The root `skills/`, `agents/`, `commands/`, and `hooks/` directories are the source of truth; do not copy their contents into platform metadata or generated output.

- Each skill is in `skills/<skill-name>/` and starts with `SKILL.md`. Put supporting documentation in `references/`, helpers in `scripts/`, structured input in `data/`, and packaged assets in `assets/`. Keep tests near the script they cover, such as `skills/databases/scripts/tests/`.
- Specialist agent definitions live in `agents/`. Claude-specific and Codex-specific agent output is generated by `install.ps1`; edit the source Markdown, never the generated files.
- Claude commands live under `commands/<namespace>/`. Codex does not support these commands, so the installer must not copy them into `.codex/`.
- Guard-rail implementation and its source configuration are `hooks/guard-rails.mjs` and `hooks/hooks.json`. The shared defaults and artifact paths are in `.hs.json`; MCP source configuration is `.mcp.json`.
- Platform manifests are `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, `.codex-plugin/plugin.json`, and `.agents/plugins/marketplace.json`. Keep names, display names, versions, descriptions, and relevant metadata aligned across them.

`install.ps1` is the only installer. It may generate `.claude/` and `.codex/` in the chosen destination; those directories are ignored by Git and must never be edited or committed.

## Development & Validation

There is no repository-wide build. Run validation proportional to the component changed:

- `python -m pytest skills/databases/scripts/tests` validates database helpers.
- `python -m pytest skills/ui-styling/scripts/tests` validates UI-styling helpers.
- `cd skills/chrome-devtools/scripts; npm test` validates Chrome DevTools scripts.
- `node --test tests/hooks/guard-rails.test.mjs` validates guard rails.
- `pwsh -NoProfile -File tests/install-smoke.ps1` validates the installer in an isolated temporary workspace.

For plugin, hook, or installer changes:

- Run `claude plugin validate . --strict` when the Claude CLI is available after changing Claude manifests, hooks, skills, agents, or commands.
- Run `python -m json.tool` for every changed JSON file. Use a Codex plugin validator when one is available.
- Run the installer smoke test. It verifies Claude and Codex output, generated hook wiring, MCP conversion, and that source agents remain unchanged.

Any future distribution build must read only the root source directories and write only to `dist/<platform>/`; never copy generated output back into source.

## Coding Style & Naming Conventions

Preserve the edited component's existing style. Use four spaces for Python, ES modules for JavaScript, and PowerShell conventions already used by `install.ps1` and `tests/install-smoke.ps1`.

Use lowercase kebab-case for skill directories, `snake_case` for Python modules, `test_<module>.py` for Python tests, and uppercase `SKILL.md` with valid YAML front matter and an `hs:`-prefixed name. Keep agent source files and command files as Markdown. Do not add platform-specific copies of source content.

## Testing Guidelines

Tests should describe observable behavior and avoid network calls, credentials, and destructive operations. Extend the relevant test suite whenever helper, hook, or installer behavior changes. Coverage JSON under skill test directories is local configuration, not a repository-wide coverage requirement.

The installer test creates and removes its own exact temporary directory; do not change it to operate on a working checkout or a broad temporary path.

## Commit & Pull Request Guidelines

Use short imperative commit subjects, for example `Add database backup validation`. Keep commits focused. Pull requests should identify the affected skill, agent, command, hook, manifest, or installer behavior; state the validation run; and link the related issue. Include screenshots or sample output for visual or browser-facing work.

## Security & Configuration

Never commit API keys, `.env` files, private keys, certificates, or user data. Document environment variables in the relevant `SKILL.md` or script README with sanitized examples. Preserve the privacy and scout guard rails unless a scoped requirement explicitly changes them.

Treat remote PowerShell installation as executable code delivery. Keep the GitHub bootstrap in `install.ps1` minimal: download only the repository archive, install to an explicit destination, and remove its exact temporary files. Document that remote users should review the script and pin a reviewed `-Ref` for production use. Do not add another installer or a separate remote bootstrap script.
